# 2026-02-27 Intent UI Route Integration V1

## Goal
- Integrate the migrated `frontend/intent-ui` app into the canonical `frontend/` surface as a first-class Edgerun route.
- Keep existing Edgerun frontend architecture intact while making Intent UI accessible without running a separate server.

## Non-Goals
- No full component-by-component rewrite of Intent UI into TypeScript pages.
- No replacement of existing Edgerun pages or nav structure beyond adding an Intent entry point.
- No port changes or separate runtime process for production serving.

## Security and Constraints
- Keep Bun-only workflows.
- Keep canonical frontend root in `frontend/`.
- Serve Intent UI assets as static files under `frontend/public/intent-ui`.
- Keep route integration deterministic and build-time generated.

## Acceptance Criteria
1. `frontend/public/intent-ui/index.html` and required bundle assets are generated by a script under `frontend/scripts/`.
2. Frontend build pipeline includes the Intent UI sync/build step before static site generation.
3. New route `/intent/` exists in static routes and client routes.
4. `/intent/` page renders an embedded Intent UI viewport and links to direct `/intent-ui/` path.
5. Main navigation includes an Intent link.
6. `cd frontend && bun run check` passes.
7. `cd frontend && bun run build` passes.

## Rollout
- Add build script + route + nav in one additive change.
- Verify local build/check.

## Rollback
- Remove Intent route/nav entries and remove `intent-ui` sync script from package build chain.
- Delete generated `public/intent-ui` assets if needed.
